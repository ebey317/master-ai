### NAVIGATION — HOW TO SCROLL UP/DOWN AND OTHER MOVES
The input box (🥷) stays pinned at the bottom. These scroll the chat
output region ABOVE it without moving your cursor.

KEYBOARD (when focus is in the input):
  Tab            — auto-complete any command from the 60+ word list
  ↑ / ↓          — scroll your input HISTORY (what you previously typed)
  Shift+↑        — SCROLL CHAT UP.   Tap = 3 lines. Hold = continuous.
  Shift+↓        — SCROLL CHAT DOWN. Tap = 3 lines. Hold = continuous.
  Ctrl+C         — interrupt the current reply + save session

TYPED WORDS (alternative to keys — works on any keyboard, any remote):
  doctor — live health/productivity check (URLs, services, models, task)
  up     — scroll up one page

  top    — jump to the very top of the chat

  copy   — push last AI reply to X11 clipboard (via xclip)

MOUSE (opt-in):

Use `refresh` after switching mouse profiles so the TUI relaunches with the
saved SENSEI_MOUSE value. Remote is best for phone scrolling/taps. Local is

"Tap vs hold" is the OS's key-repeat. One press fires once. Hold fires
~30/sec for as long as you hold. Useful with Shift+↑/↓.

### COPY / PASTE IN SENSEI
Because the TUI captures mouse events for scrolling, plain click-drag does
NOT select text — the terminal passes the click to the app. Two ways to copy:

  1. SHIFT + click-drag over the text you want → terminal selects it
     (Shift overrides the app's mouse capture — this is a gnome-terminal /
     xterm convention). Then Ctrl+Shift+C to copy to clipboard.

  2. Type `copy` (or `clip`) at the Sensei prompt — pushes the LAST AI reply
     straight into the X11 clipboard via xclip. Fast path for "I want that
     whole answer."

To temporarily disable mouse capture (and get back classic click-drag),
launch with SENSEI_MOUSE=0: `SENSEI_MOUSE=0 bash ~/scripts/launch_master_ai.sh`

## RECOVERY — if Master AI freezes, glitches, or won't respond
Escalate from gentle to aggressive. Each layer does more than the last.

  1. inside Master AI:  refresh                — soft re-exec, same process
  2. inside Master AI:  kick                   — exit 42, supervisor respawns in 3s
  3. any shell:         ~/scripts/master_ai_refresh.sh   — kill engine, supervisor respawns
  4. any shell:         ~/scripts/master_ai_kick.sh      — kill tmux session + relaunch
  5. any shell:         pkill -KILL -f "python3.*master_ai.py"
                        — FORCE-kill the Python engine process directly.
                        Use when refresh/kick commands aren't responding.
                        Supervisor loop will auto-restart within 3 seconds.
  6. any shell:         tmux kill-session -t master-ai && bash ~/scripts/launch_master_ai.sh
                        — nuke + start over (full rebuild).

## MANUAL REBOOT — if Sensei exited cleanly and won't auto-restart
If you see "Master AI exited cleanly." at your shell prompt, the supervisor
loop intentionally broke out (that happens on exit 0 — e.g. after typing 'x',
or after a clean shutdown). The tmux session is still alive; you just need to
relaunch the engine. Pick ONE of these and run it in any shell:

  A. PREFERRED — reattach + relaunch engine inside the existing session:
       bash ~/scripts/launch_master_ai.sh
     The launcher detects tmux is still up, sends the supervisor command back
     in, and attaches you. This is the normal restart path.

  B. Classic I/O fallback (if the prompt_toolkit TUI is misbehaving and you
     want the plain readline input back for this session):
       SENSEI_TUI=0 bash ~/scripts/launch_master_ai.sh
     Set SENSEI_TUI=0 in your shell env to persist across restarts.

  C. Full nuke if tmux itself is wedged:
       tmux kill-session -t master-ai
       bash ~/scripts/launch_master_ai.sh

  D. Start from the master menu instead:
       bash ~/scripts/master.sh   → option 4 (Sensei tmux AI)
     Option 4 calls launch_master_ai.sh under the hood — same effect as A.

If 'refresh' inside Sensei no longer works, step up to 'kick' (exit 42 forces
the supervisor to respawn). If even that fails and you're stuck at the shell,
follow steps A or C above.

Crash log (if non-zero exit): ~/scripts/master.crash.log
Engine PID check:             pgrep -af "python3.*master_ai.py"
